양방향 영상출력을 해보자

회사에서 영상관련 프로젝트로 인해 ffmpeg와 webrtc를 처음 접하게 되었습니다. WebRTC 내용이 워낙에 어렵고 레퍼런스도 적은데다 기술 발표자료가 즐비할 뿐 영상강좌는 외국 분들이 올린 것이 전부였습니다.

물론 더 찾아보면 있을 수 있지만 단기간에 빨리 익혀야하는게 슬픈 현실...

이것저것 참고한 것들과 작업한 결과물과 작업하면서 겪은 오류 상황 및 문제해결을 기록하려합니다.

WebRTC API를 들여다 보자

이전 포스팅에서 react에서 webrtc를 사용해서 간단한 영상출력까지 해보았습니다. 이번에는 채팅룸을 만들어 상대방이 접속하면 서로 영상을 출력하는 것을 구현하려합니다.

먼저 알아야하는 것은 WebRTC API에 관한 내용인데요, WebRTC API는 그렇게 많지 않고 MDN web docs에 설명이 굉장히 잘 되어 있습니다. 다만, 처음 접할 때는 당연히 잘 정리된 말도 어렵게 느껴지기 마련이지요.

포스팅 내용이 주관적인 공부 패턴을 기준으로 하기 때문에 내용이 뒤죽박죽이어도 양해바랍니다 🥲

RTCPeerConnection

제일 중요한 인터페이스라고 말하고 싶은 녀석입니다. RTCPeerConnection 생성자를 통해 만든 인스턴스를 자주 다룰 것 입니다. 제일 핵심이라고 해도 과언이 아니라 생각합니다.

1const pc = new RTCPeerConnection();

위와 같이 생성자를 통해 인스턴스를 만드는데요, 이때 pc 변수는 datachannel이나 이벤트, icecandidate를 추가 하는 등의 행위를 하게 해주는 핵심적인 기능을 담당합니다. WebRTC는 시그널링 서버를 필요로 하는데 시그널링을 지원하지 않으며, socket.io라는 WebSocket library를 사용해야 합니다.

미디어를 주고 받는 흐름을 알면 나중에 어떤 방식으로 무엇부터 해야하는지 감이 잡히기 때문에 흐름도를 보겠습니다.

순서를 나열해보면 다음과 같습니다.

  1. 유저가 접속한다.
  2. getUserMedia로 카메라 권한 요청을 한다.
  3. peerConnection에서 localDescription을 작성한다.
  4. dataChannel 생성
  5. createOffer
  6. sendOffer
  7. socket에서 다른 사용자에게 broadcast로 offer를 전달한다.
  8. 받은 offer를 remoteDescription에 넣는다.
  9. createAnswer
  10. localDescription을 작성한다.
  11. sendAnswer
  12. socket에서 다른 사용자에게 broadcast로 answer를 전달한다.
  13. 받은 answer를 remoteDescription에 넣는다.
  14. iceCandidate 이벤트가 발생하는 영역에서 socket에 iceCandidates를 전송한다.
  15. onTrack 이벤트가 발생하고, 해당 트랙을 remoteVideo의 srcObject에 넣는다.
  16. 다른 사용자 또한 iceCandidates 이벤트 발생 시 상대방에게 iceCandidates를 전송한다.
  17. 결과적으로 서로 영상 교환이 되어 양방향 출력이 된다.

아직 정리가 완전히 된 내용이 아니기 때문에 자세한 자료는 공식 문서를 참조하시는 것을 권장드리며, 개념적으로 보자면 서로 하나씩 교환한다고 생각하시는 게 이해를 도울 수 있을 것 같습니다.

offer가 만들어지고 전송되면 answer로 응답해주고, 각자 localDescription, remoteDescription이 작성되면 이후 iceCandidate를 주고 받습니다.

그 과정에서 onTrack 이벤트를 설정해서 각 remoteVideo에 출력시키는 흐름입니다.

1// src/modules/room.js
2const ADD = "room/ADD";
3const READ = "room/READ";
4const SAVE = "room/SAVE";
5const REMOVE = "room/REMOVE";
6
7export const add = (room) => ({ type: ADD, room });
8export const read = () => ({ type: READ });
9export const save = () => ({ type: SAVE });
10export const remove = (room) => ({ type: REMOVE, room });
11
12const initialState = [];
13
14export default function room(state = initialState, action) {
15  switch (action.type) {
16    case ADD:
17      if (state.indexOf(action.room) === -1) {
18        return state.concat(action.room);
19      } else {
20        return state;
21      }
22    case READ:
23      const webrtcRoom = () => localStorage.getItem("webrtcRoom");
24      if (!webrtcRoom()) localStorage.setItem("webrtcRoom", "[]");
25      return JSON.parse(webrtcRoom());
26    case SAVE:
27      localStorage["webrtcRoom"] = JSON.stringify(state);
28      return state;
29    case REMOVE:
30      return state.filter((_) => _ !== action.room);
31    default:
32      return state;
33  }
34}

앞서 흐름도에서 봤던 이미지를 참고하면서 만들면 더욱 좋습니다. 이제부터 소켓서버와 통신하면서 icecandidate나 offer, answer를 주고 받으면서 track을 얻어서 remoteVideo에 stream을 저장하여 상대 영상을 띄우겠습니다. 큰 골자는 영상 데이터를 주고받는 것입니다.

20

22      // 서버 연결 표시
23      socket.emit("join", params.id);
24      dispatch(add(params.id));
25      dispatch(save());
26
27      init();
28
29      // 서버에서 온 메세지
30      socket.on("message", (msg) => {
31        switch (msg.type) {
32          case "offer":
33            pc.setRemoteDescription(new RTCSessionDescription(msg))
34              .then(() => pc.createAnswer())
35              .then(setDescription)
36              .then(sendDescription)
37              .catch(handleError);
38            break;
39          case "answer":
40            pc.setRemoteDescription(new RTCSessionDescription(msg));
41            break;
42          case "candidate":
43            pc.addIceCandidate(msg.candidate);
44            break;
45          default:
46            return;
47        }
48      });
49
50      socket.on("userInRoom", (msg) => {
51        console.log("[인원 변동] 현재 인원 ", msg);
52        saveCounter = msg;
53        setCountUser(msg);
54      });
55    });
56
57    // 카메라 권한 요청
58    // getUserMedia ...
59
60    return () => {
61      socket.disconnect();
62    };
63  }, []);
64
65  function init() {
66    // peerConnection 생성
67    pc = new RTCPeerConnection({
68      iceServers: [{ urls: "stun:stun.l.google.com:19302" }],
69    });
70
71    pc.onicecandidate = function (e) {
72      console.log("onicecandidate", e);
73      if (e.candidate) {
74        socket.send({
75          type: "candidate",
76          candidate: e.candidate,
77        });
78      }
79    };
80
81    pc.ontrack = function (e) {
82      console.log("ontrack", e);
83      remoteVideo.current.srcObject = remoteStream = e.streams[0];
84    };
85
86    pc.ondatachannel = function (e) {
87      dataChannel = e.channel;
88      console.log(localStream.getVideoTracks());

추가된 함수들은 지금부터 흐름을 하나 씩 이야기 할 것이기 때문에 굳이 설명은 하지 않겠습니다. 앞서 getUserMedia보다 선두로 socket을 연결하고 connect된 후로 message, userInRoom 이벤트를 수신하고 있습니다.

message는 offer, answer, candidate를 수신하고 broadcast로 본인 외 모두에게 데이터를 주는 이벤트입니다.

client의 message 이벤트는 서버에서 온 msg를 받고 switch문으로 msg의 type을 분기합니다.

offer면 remoteDescription - answer - localDescription 순으로 생성하고 생성된 answer를 message이벤트로 다시 서버에 전송합니다.

그러면 서버는 다시 client에게 message로 emit하고, 다른 사용자들은 message이벤트가 발생해서 answer를 전달 받아 peerConnection의 remoteDescription에 sdp를 저장합니다.

그리고 마지막으로 candidate 분기는 onicecandidate 이벤트가 발생해야 작동하는 분기인데, 인원 상관없이 icecandidate 이벤트는 발생합니다.

인원이 둘 이상이어야 서로 icecandidate를 교환하고 각자의 peerConnection에 candidate를 추가합니다.

그렇게 모든 교환이 이루어지고나면 getUserMedia를 통해 카메라 장치를 요청하고 해당 stream을 video 태그에 전달해주고, peerConnection에 track을 추가하면서 ontrack이벤트 발생 시 서로 교환 받은 stream데이터를 remoteVideo에 전달합니다.

다소 부정확할 수 있으나 위에서 이야기한 교환 순서나 작동 방식의 큰 흐름과 골자는 크게 흔들리지 않는다고 생각합니다.

나머지 함수는 위의 골자대로 생각하면서 보면 금방 알아채시리라 생각합니다.

dondido님의 16버전 소스코드에 처음 본 redux까지 섞여있어서 많이 난해했습니다. WebRTC도 생소했고, react를 시작한지 이제 2달이라 하기가 싫었습니다. 힘들었습니다.
